# FuzeKeys Deployment (GitOps)

GitOps deploy wiring for **FuzeKeys** onto the shared **FuzeInfra Contabo k3s**
cluster. Everything here is a **definition** — there is no live cluster in this
repo. **Prod is GitOps: ArgoCD syncs `deploy/` from `master`; nothing is
hand-deployed to prod.** Local/dev is Helm/Skaffold on kind.

## Layout

```
deploy/
├── README.md                       # this file
├── argocd/
│   ├── project.yaml                # AppProject "fuzekeys" (sourceRepos + destinations)
│   ├── app-of-apps.yaml            # root Application -> deploy/argocd/applications (recurse)
│   ├── applications/
│   │   ├── fuzekeys-platform.yaml  # Helm Application -> deploy/helm/fuzekeys (values-contabo)
│   │   └── fuzekeys-sealed.yaml    # Application -> deploy/argocd/sealed (SealedSecrets)
│   └── sealed/                     # SealedSecrets (ciphertext only) — generated by seal-secrets.yml
├── helm/fuzekeys/                  # the workload Helm chart
│   ├── Chart.yaml                  # appVersion == image tag (release/tag-bump target)
│   ├── values.yaml                 # defaults + fuzeinfra.* contract + per-component `enabled` gates
│   ├── values-contabo.yaml         # ArgoCD-applied prod overlay (traefik, no TLS, MVP gates)
│   └── templates/                  # Deployments/Services/Ingress/CM/Secret/PVC/Jobs
└── terraform/
    ├── cloudflare-dns.tf           # product DNS (keys / api.keys) — APPLIED BY FUZEINFRA
    └── variables.tf                # token + zone id (injected at apply; never committed)
```

## Argo wiring (hybrid Argo)

FuzeKeys is a single independently-lifecycled product, so it gets its **own**
Argo project + app-of-apps (not folded into the FuzeFront umbrella):

1. Register **`deploy/argocd/app-of-apps.yaml`** in Argo once (and the AppProject
   `deploy/argocd/project.yaml`). Argo then discovers the child Applications in
   `deploy/argocd/applications/` (recurse) and reconciles them:
   - **`fuzekeys-platform`** — the Helm chart into namespace `fuzekeys`
     (`automated{prune,selfHeal}`, `CreateNamespace`, `ServerSideApply`, retry backoff).
   - **`fuzekeys-sealed`** — the SealedSecrets in `deploy/argocd/sealed/`
     (`prune:false` — never auto-prune secrets).
2. All Applications track **`targetRevision: master`** (this repo's default branch).

## Consuming FuzeInfra (one-way dependency)

FuzeInfra is **shared infrastructure** (Postgres/Redis) owned by another team.
FuzeKeys is a **consumer** and the dependency is strictly one-way: it assumes
only FuzeInfra's published Service contract — `fuzeinfra-<svc>.<ns>.svc.cluster.local`.

| Input | Default | Meaning |
|-------|---------|---------|
| `fuzeinfra.namespace` | `fuzeinfra` | Namespace FuzeInfra publishes Services into. |
| `fuzeinfra.useNamespacedDns` | `true` | Build the in-cluster FQDN (contract / main path). |
| `fuzeinfra.postgres.host`/`port` | `fuzeinfra-postgres`/`5432` | → `fuzeinfra-postgres.fuzeinfra.svc.cluster.local:5432` |
| `fuzeinfra.postgres.database` | `fuzekeys` | FuzeKeys-owned DB (provisioned by FuzeInfra's DB onboarding). |
| `fuzeinfra.redis.host`/`port` | `fuzeinfra-redis`/`6379` | → `fuzeinfra-redis.fuzeinfra.svc.cluster.local:6379` |

Credentials are **never** in ConfigMaps and have **no** baked defaults — they
come from the `fuzekeys-secrets` SealedSecret (`secrets.existingSecret`). The
connection URLs (`DATABASE_URL` / `DATABASE_URL_ASYNC` / `REDIS_URL`) are composed
from those creds + the host/port contract.

> The FuzeKeys Postgres role + `fuzekeys` database are **not** created here — that
> is FuzeInfra's DB-onboarding (delegated to FuzeInfra via `@claude`).

## Components + `enabled` gates

| Component | MVP (values-contabo) | Notes |
|-----------|----------------------|-------|
| `backend` | **on** | FastAPI, :8002, `/health`; envFrom config + secret |
| `migrate` | **on** | `alembic upgrade head` (Helm pre-install/pre-upgrade hook → Argo PreSync) |
| `frontend`| **on** | React/nginx; container :3000, Service :80 |
| `tokenizer`, `presidio`, `litellm`, `vault`, `vaultBootstrap`, `tokenizerRedis` | **off** | Authored in full; **FOLLOW-UP: enable after Vault-in-k8s auto-unseal design + security review** |
| `ollama` | **off (always)** | GPU/heavy |

Every container declares resource **requests AND limits** (shared-cluster rule).
App pods pin to the workload node via `nodeSelector: node-role=workload` when
`global.nodeRole` is set (values-contabo sets `workload`).

## Ingress / TLS

Traefik behind the shared **Cloudflare Tunnel**. CF terminates TLS at the edge and
Traefik is ClusterIP, so the chart has **no cert-manager annotation and no `tls:`
block** by contract. Hosts:

- `keys.prod.fuzefront.com` → frontend :80
- `api.keys.prod.fuzefront.com` → backend :8002

DNS (`deploy/terraform/cloudflare-dns.tf`) declares both as proxied CNAMEs to the
tunnel — **applied by FuzeInfra's handler**, not from this repo.

## Secrets (SealedSecrets)

`.github/workflows/seal-secrets.yml` (manual `workflow_dispatch`, idempotent):

- Fetches FuzeInfra's **public** sealed-secrets cert (no cluster access, no
  private key), builds Secrets in-memory, `kubeseal`s them into
  `deploy/argocd/sealed/`, and **opens a PR** (signing-safe — the protected
  `master` is never pushed directly). Argo syncs them on merge.
- Seals: `ghcr-pull-secret`, `fuzekeys-secrets` (composed `DATABASE_URL` +
  generated `SECRET_KEY`/`MASTER_KEY_SALT`/`CREDENTIAL_ENCRYPTION_KEY`), and
  `fuzekeys-tokenizer-secrets` (scaffold).
- **`CREDENTIAL_ENCRYPTION_KEY` is a Fernet key** (urlsafe-base64), NOT hex — the
  backend validates it as a Fernet key and refuses to start otherwise.

`ghcr-pull-secret` must match the pod `imagePullSecrets` ref exactly.

## Images / release

`.github/workflows/build-and-push.yml` builds `backend`, `frontend`, and
`pii-tokenizer` to `ghcr.io/izzywdev/fuzekeys/<component>`, tagged
`:latest`, `:<Chart.appVersion>`, and `:<sha>`. The **release / tag-bump** is:
bump `Chart.yaml appVersion` (+ the CI matrix tags it) and let `image.tag` default
to it; commit the bump → Argo syncs. Never hand-deploy the bump to prod.

## Validation (definitions-only)

`.github/workflows/deploy-platform.yml` runs on `deploy/**` changes:
`helm lint` + `helm template` + **kubeconform** (`-ignore-missing-schemas`, both
MVP and full-stack renders) + **actionlint**. No live deploy — Argo does that.

Locally:

```bash
helm lint deploy/helm/fuzekeys -f deploy/helm/fuzekeys/values-contabo.yaml
helm template deploy/helm/fuzekeys -f deploy/helm/fuzekeys/values-contabo.yaml \
  | kubeconform -ignore-missing-schemas -summary
```

## Follow-ups (deliberate deferrals)

- **PII-tokenizer stack (Vault-in-k8s):** Vault is sealed on every restart and the
  current design keeps the unseal key in a local volume — not acceptable for k8s
  prod. The stack stays `enabled:false` until auto-unseal (KMS/transit or an
  operator) is designed and the LLM data-path is security-reviewed. The shared
  Vault init-file PVC is RWO (single-node) — part of the same redesign.
- **FuzeInfra DB onboarding:** the `fuzekeys` Postgres role + database are
  provisioned by FuzeInfra (delegated).
- **Argo registration + DNS apply:** registering the app-of-apps and applying
  `cloudflare-dns.tf` happen on the FuzeInfra/Argo side after merge.
